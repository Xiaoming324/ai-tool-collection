## 阶段 0 — 脚手架 & Claude 对话基座

目标：项目能起来，能跟 Claude 流式对话。

- ✅ 环境就绪（JDK 21、Maven、本地 MySQL、Docker 起 Redis Stack、密钥放 `application-local.yaml`）；文件存储用真 AWS S3（阶段 3 接入）
- ✅ `pom.xml` 加 `spring-ai-starter-model-anthropic`（chat）+ 保留 `openai` starter（后续 embedding）
- ✅ `pom.xml` 加 lombok（不写版本，交给 Spring Boot 父 POM 管理；`provided` 作用域）
- ✅ `<java.version>` 设为 21
- ✅ `application.yaml`：Claude `claude-sonnet-4-6` 做对话（开发期省钱；上线可换 `claude-opus-4-8`）
- ✅ `CommonConfiguration`：`ChatClient` bean（显式注入 `AnthropicChatModel` + 系统提示 + 20 条滑动窗口记忆）
- ✅ `ChatController`：流式 `/ai/chat`（`prompt` + `chatId`）
- ✅ 跑通验证：中文回复正常、会话记忆生效、确认走 `claude-sonnet-4-6`
- ✅ 基础包名统一为 `com.itheima.ai`（对照 heima-ai 迁移更方便）

> **关于 CORS**：不在后端配。开发期用前端 dev server 代理（如 Vite `server.proxy` 把 `/api` 转发到 `localhost:8080`），生产期用 nginx 把 `/api` 反代到后端——前后端同源，不触发跨域。只有前端"直连不同源后端且不走代理"时才需后端 CORS，本项目不走这条路。

## 阶段 1 — 用户管理 + JWT 鉴权（模块④）

目标：注册/登录拿 token，之后所有 `/ai/*` 接口要校验 token 并取出 `userId`。

- ✅ 加依赖：`spring-boot-starter-security`、`jjwt`（api/impl/jackson 0.12.6）、`mybatis-plus-spring-boot4-starter` 3.5.15
- ✅ `application.yaml` 补 datasource（MySQL url/username/driver；密码在 local）
- ✅ 建用户表 `user`（id、username、password、created_at）
- ✅ `User` 实体 + `UserMapper`(BaseMapper) + `IUserService`/`UserServiceImpl`(IService)；入口类加 `@MapperScan`
- ✅ `JwtUtil`：用 `app.jwt.secret` 签发/解析 token（jjwt 0.12.x，HS256；有效期 1 天）
- ✅ `SecurityConfig`：`BCryptPasswordEncoder` bean + 过滤器链（放行 `/auth/**`、其余 authenticated、无状态、关 CSRF）
- ✅ `/auth/register`：校验重名，BCrypt 加密存库；`/auth/login`：校验密码 → `JwtUtil` 签发 token
- ✅ 入参校验（`@Valid` + DTO 注解：非空、无空格、长度）+ 统一响应 `Result<T>` + 全局异常处理器（`@RestControllerAdvice`：校验/业务错误→400，兜底 `Exception`→500 并 `log.error` 记堆栈）
- ✅ `JwtAuthFilter`(`OncePerRequestFilter`)：校验 token → 把 userId 放进 `SecurityContextHolder`，`addFilterBefore` 挂进链；401 认证入口
- ✅ 验证：无 token→401、错误 token→401、带有效 token→200 且 Claude 正常回复

## 阶段 2 — AI 对话强化（模块②）：多模态 + 持久化记忆

目标：对话支持图片；模型记忆持久化到 MySQL；业务侧会话/文本消息独立落库，按用户隔离。

- ✅ 记忆持久化：加 `spring-ai-starter-model-chat-memory-repository-jdbc`；`application.yaml` 开 `spring.ai.chat.memory.repository.jdbc.initialize-schema: always` 自动建 `SPRING_AI_CHAT_MEMORY` 表
- ✅ `CommonConfiguration` 的 `chatMemory()` 注入 `ChatMemoryRepository`（Jdbc 版）替代 InMemory；已验证重启后记忆不丢
- ✅ 记忆 key 从 `chatId` 升级为 `userId + ":" + chatId`（`ChatController` 用 `@AuthenticationPrincipal` 取 userId）；已验证两用户同 chatId 互不可见
- ✅ 多模态：`/ai/chat` 接收图片（`MultipartFile`→`Media`），`textChat`/`multiModalChat` 分流，传给 Claude Vision；已验证识图（"PURPLE 7"）
- ✅ 修复：流式(Flux)走异步 dispatch 被 Security 二次鉴权拦截 → `SecurityConfiguration` 放行 `DispatcherType.ASYNC/FORWARD`（初次 REQUEST 仍鉴权）
- ✅ 会话表升级为 `chat_session`：记录 `(userId,type,chatId,title,updatedAt)`；`type` 用枚举 `ChatType`(`@EnumValue`+`@JsonValue`+路径 `Converter`)；`/ai/history/{type}` 返回 `id + chatId + title`
- ✅ 文本消息独立落库：新增 `chat_message` + `MessageRole`，`/ai/chat` 在调用 Claude 前保存 `user` 消息，流式完成后保存 `assistant` 消息
- ✅ 历史消息接口切换：`/ai/history/{type}/{chatId}` 不再读取 `ChatMemory` 作为前端消息来源，而是按 `chat_session -> chat_message` 返回业务消息
- ✅ 验证：发图片能描述内容；重启后记忆与历史仍在；A/B 用户互相看不到
- ✅ 记忆 key 含 type：`conversationId = userId:type:chatId`（避免不同类型复用同 chatId 时记忆串台）；`/ai/history/{type}/{chatId}` 同步用 type 拼 key
- ✅ 非法 type 路径 → `MethodArgumentTypeMismatchException` handler 返回 400

## 阶段 3 — ChatPDF（模块①）：RAG + 文件存储

目标：上传 PDF → 切块向量化入库 → 针对文档提问，Claude 基于检索原文作答。

- ✅ 文件层前置：补 `stored_file`（文件元数据）+ `chat_message_attachment`（消息附件关系），让图片/PDF 能和具体消息绑定
- ✅ 加依赖：`spring-ai-pdf-document-reader`、`spring-ai-starter-vector-store-redis`、`spring-ai-advisors-vector-store`
- ✅ 启用 Redis 向量库（使用 Redis Stack，补齐 metadata schema，未沿用 heima-ai 里的排除配置）
- ✅ `application.yaml` 配 openai-compat 指向 DashScope 的 **embedding**（`text-embedding-v4`，1024 维）+ Redis 连接
- ✅ 文件存储：加 `software.amazon.awssdk:s3`（AWS SDK v2），配真 AWS S3（region/bucket/凭证放 `application-local.yaml`），写 `FileStorageService`(上传 + 生成预签名 URL)
- ✅ 上传 `/ai/pdf/upload/{chatId}`：已完成“原文件传 S3 + 元数据入 MySQL + 创建 PDF 会话 + PDF 解析/按页切分 + embedding + 存 Redis”
- ✅ 取文件 `/ai/pdf/file/{fileId}`：校验归属 → 返回 S3 **预签名 URL**（前端 iframe/pdf.js 加载）
- ✅ RAG 问答 `/ai/pdf/chat`：`QuestionAnswerAdvisor` 检索（当前 topK=2、similarityThreshold=0.5），metadata 按 `user_id + session_id` 过滤
- ✅ 验证：上传 PDF → 问答 → 多轮追问 → 消息落库 → 会话隔离，主链路已跑通
