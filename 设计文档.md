# AI Tool Collection —— 基于 Claude 的多功能 AI 工具集 · 设计文档

> 产品名：**AI Tool Collection**。界面全英文；与模型对话时由用户决定回复语言（让 Claude 用中文或英文回答均可）。

## Context（背景与目标）

现有目录下有两个项目：
- `heima-ai/`：黑马 Spring AI 教程后端（Spring Boot 4.0.6 + Spring AI 2.0.0-M8），对话和向量化**都用阿里云 DashScope（Qwen + text-embedding-v4）**，向量库是内存版 `SimpleVectorStore`，含 ChatPDF / 多模态对话 / 智能客服(工具调用) / 哄哄模拟器 四个模块。
- `spring-ai-nginx/`：编译后的 Vue 前端 + nginx，代理 `/api` 到后端 `localhost:8080`。

**目标**：重新做一个类似 ChatPDF 的项目，把 **Function Calling、RAG、AI 对话、图片多模态** 这几个能力用 **Claude** 实现。

**关键事实（决定架构的前提）**：Claude 官方 Messages API
- ✅ 原生支持：文本、**图片**、**PDF 文档**
- ❌ 不提供 **embedding（向量化）接口** → RAG 的向量化必须由第三方模型完成
- ❌ 不接收 **音频 / 视频** 输入 → 本项目据此**不做音视频**，聚焦文字/图片/PDF

因此新项目天然是"**对话用 Claude、向量化用阿里**"的组合，这是与原项目（阿里一家包揽）的最大不同。

## 产品定位

**Claude 驱动的多功能 AI 助手平台**，核心是 ChatPDF 文档问答，并用三个模块分别承载三大技术亮点。

### 功能模块（3 个 AI 模块 + 1 个用户管理，已去掉角色扮演）

| 模块 | 作用 | 技术亮点 |
|------|------|----------|
| **① ChatPDF（文档问答）** | 上传 PDF → 切块/向量化入库 → 针对文档提问，Claude 基于检索原文作答；左侧 PDF 预览右侧对话，支持多文档与历史 | **RAG 全流程** |
| **② 智能对话（多模态聊天）** | 通用文字对话 + **图片理解**（发图让 Claude 分析）；流式输出 + 对话记忆 | **Claude Vision 多模态** |
| **③ 旅游推荐助手（工具调用）** | Claude 推荐目的地/景点/行程，并自主调用工具接真实 API（天气、景点、汇率、国家信息）+ 收藏行程落库 | **Function Calling / Tool Use** |
| **④ 用户管理（注册/登录/鉴权）** | 注册后才能使用；登录发 JWT；所有 AI 数据按用户隔离，每个用户有独立的会话与文档 | **Spring Security + JWT + 数据隔离** |

> 范围说明：**不做音视频**（Claude 不原生支持音频/视频；本项目聚焦文字、图片、PDF）。多模态仅指文字 + 图片。

## 技术架构

```
React 前端  ──(REST + SSE 流式)──▶  Spring Boot + Spring AI 后端
                                          │
        ┌─────────────────────────────────┼──────────────────────────────┐
        ▼                                 ▼                                ▼
  Claude(对话/工具/图片)          阿里 embedding(向量化)              Redis Stack(向量库)
  spring-ai-starter-model-       text-embedding-v4                  Docker 一条命令起，持久化
  anthropic / claude-opus-4-8    (经 OpenAI 兼容模式接入)            RediSearch 向量检索 + RedisInsight
        │                                                                  ▲
  用户/会话历史/对话记忆/业务数据 ──▶ MySQL（持久化，重启不丢，按 userId 隔离）  │
  鉴权: Spring Security + JWT（注册/登录后才能用 AI 接口）                  │
  上传的 PDF/图片原文件: AWS S3 私有桶 uploads/{userId}/ + 预签名URL显示 ───┘
  (直接用真 AWS S3；元数据存 MySQL)
```

### 技术选型与理由

| 关注点 | 选型 | 理由 |
|--------|------|------|
| 后端框架 | Spring Boot + Spring AI | 复用现有 Java 栈与 controller/RAG 结构，迁移成本最低 |
| 对话模型 | **Claude `claude-opus-4-8`** | 通过 `spring-ai-starter-model-anthropic` 接入；强在文本推理、工具调用、图片理解 |
| 向量化 | **阿里 `text-embedding-v4`** | 多语言（中英文都强）、国内访问快、便宜、沿用现有；与 Claude 不冲突 |
| 向量库 | **Redis（Redis Stack）** | 用户已有 Docker，一条命令起库；持久化、RediSearch 向量检索、自带 RedisInsight 可视化；原 `pom.xml` 已含 `spring-ai-starter-vector-store-redis` 依赖，启用即可 |
| 前端 | **React** | 用户指定；调用后端 REST + SSE 流式接口 |
| 鉴权 | **Spring Security + JWT** | 注册/登录后发 token，AI 接口校验后拿 userId 做数据隔离；密码 BCrypt 加密 |
| 用户/历史/记忆/业务数据 | MySQL | **持久化**：用户表、会话表 `chat_session`、消息表 `chat_message`、对话记忆（`JdbcChatMemoryRepository`）、工具调用业务数据、上传文件元数据；重启不丢，按 userId 隔离 |
| 上传文件原文件 | **AWS S3**（私有桶 + 预签名 URL） | 对象存储标准方案，可扩展；私有桶+时效签名 URL 保证按用户隔离且省后端带宽；用 **AWS SDK v2** 接入（不用 Spring Cloud AWS，避开 Boot 4 兼容风险）；**直接用真 AWS S3**（开发期成本极低/近免费，新账号 12 个月免费套餐） |

> **向量库说明**：用 Redis Stack（不是普通 Redis——普通 Redis 无向量检索，需 RediSearch 模块）。Docker 起库：`docker run -d --name redis-stack -p 6379:6379 -p 8001:8001 redis/redis-stack:latest`。后续如需更强过滤/规模可平滑切 PGVector/Qdrant（Spring AI 换向量库只是换一个 VectorStore bean，业务代码不变）。

## 关键设计点

### 1. RAG 流程（ChatPDF 核心）
- **入库**：上传 PDF → Spring AI `PagePdfDocumentReader` 解析 → 文本切块 → 阿里 `text-embedding-v4` 向量化 → 存入 Redis 向量库（`RedisVectorStore`），metadata 带 `chatId`/`filename` 以隔离不同文档；Redis 自带持久化，重启不丢。
- **问答**：用户问题 → 同模型向量化 → 相似度检索 Top-K（如阈值 0.5 / topK 2~4）→ 把检索原文作为上下文 → Claude 生成答案。
- Spring AI 用 `QuestionAnswerAdvisor` 串联检索与增强生成（参考原 `PdfController`）。

### 2. Function Calling（旅游推荐助手）
- 用 Spring AI 工具机制（`@Tool` 注解的方法 / `ToolCallback`），Claude 自主决定调用工具，把"推荐"接上真实数据与动作。**推荐本身靠 Claude 自身旅游知识**，工具用于实时数据 + 落地动作。
- **工具集（全部基于免费 API）**：
  - `searchAttractions(city)` — OpenTripMap（免费 Key）：真实景点/坐标
  - `getDestinationGuide(place)` — Wikivoyage/维基（无需 Key）：目的地攻略文字
  - `getWeather(city, dates)` — Open-Meteo（无需 Key）：最佳出行时间/穿衣
  - `getCountryInfo(country)` — REST Countries（免费）：货币/语言/基础信息
  - `getExchangeRate(from, to)` — Frankfurter（无需 Key）：预算换算
  - `saveItinerary(...)` / `listMyTrips()` — **MySQL**：收藏/保存行程（CRUD 落库 + 历史，按 userId 隔离）
- **不做**真实机票/酒店预订（这类 API 收费/受限）；聚焦推荐 + 攻略 + 行程规划 + 收藏。
- 系统提示词：约束助手只做旅游相关、用与用户相同语言回答、缺信息时反问、防注入。
- 国内城市景点 API 覆盖一般，以 Claude 知识为主 + 天气/攻略工具补充。

### 3. 多模态（智能对话）
- 文字对话 + 图片：图片以 Spring AI `Media` 对象随消息传给 Claude（base64/字节），Claude 原生 Vision 理解。
- 不做音视频。

### 3.1 会话与消息建模（当前后端主线）
- **会话层**：独立 `chat_session` 表，记录 `userId + type + chatId + title + updatedAt`；`type` 对应三种模块：`chat / pdf / travel`。前端会话列表读这张表，不再把“历史”概念混在消息表里。
- **消息层**：独立 `chat_message` 表，记录 `sessionId + userId + role + sequenceNo + textContent + createdAt`。`role` 使用枚举：`user / assistant / system / tool`。
- **职责分离**：
  - `chat_session`：给前端列会话、做标题、做最近活跃排序。
  - `chat_message`：给前端列消息、记录业务上的真实聊天内容。
  - `SPRING_AI_CHAT_MEMORY`：继续只给模型做上下文记忆，不再作为前端历史消息来源。
- **当前聊天流程**：
  - 用户请求 `/ai/chat`
  - 后端先 `createOrUpdateSession`
  - 再写入一条 `user` 消息到 `chat_message`
  - Claude 流式回复给前端
  - 流结束后聚合完整回复，再写入一条 `assistant` 消息到 `chat_message`
- **当前历史接口**：
  - `/ai/history/{type}`：返回 `chat_session`
  - `/ai/history/{type}/{chatId}`：按 `chat_session -> chat_message` 返回业务消息
  - 不再从 `ChatMemory` 读给前端

### 3.2 上传文件的存储与显示（AWS S3）
- **物理存储**：原始文件（PDF、聊天图片）存 **AWS S3 私有桶**，按用户分 key：`uploads/{userId}/{fileId}.{ext}`；MySQL 存元数据（fileId、原文件名、userId、sessionId、contentType、大小、S3 key、上传时间）。
- **消息附件关系**：文件不会直接塞进 `chat_message`；后续补 `stored_file`（文件元数据）+ `chat_message_attachment`（消息和文件关系）。这样一条消息可以挂多张图片，PDF 也能作为会话/消息资源复用。
- **向量数据**：另存 Redis 向量库（见 RAG），与原文件分开。
- **前端如何显示（关键）—— 私有桶 + 预签名 URL**：
  - 上传：前端 POST 文件 → 后端用 AWS SDK 上传到 S3 → 写 MySQL → 返回 `fileId`。
  - 显示：前端请求 `GET /ai/pdf/file/{fileId}`（PDF）或 `GET /ai/files/{fileId}`（图片）→ 后端**先校验文件归属当前 userId** → 用 `S3Presigner` 生成**有时效的预签名 URL**（如 5 分钟）返回 → 前端用该 URL 直接从 S3 加载（PDF 用 iframe/pdf.js，图片用 `<img>`）。
  - 好处：私有桶无签名打不开且 URL 会过期（安全）、浏览器直连 S3 不占后端带宽、签名仅鉴权后生成（按用户隔离）。
- **接入**：**AWS SDK v2**（`software.amazon.awssdk:s3`），自己配 `S3Client` + `S3Presigner` bean；凭证/region/bucket 放 `application-local.yaml`（不进 git）。
- **存储**：**直接用真 AWS S3 私有桶**（不本地 MinIO）。新账号 12 个月免费套餐基本覆盖学习用量，超出后也仅几分钱/月。
- **区域**：用户在美国，选就近区域（如 `us-east-1`），访问无延迟/合规问题。
- **聊天图片**：默认持久化到 S3 以支持图文历史回看；仅当次分析不回看的可只传 base64 给 Claude 不落盘。

### 4. 流式、记忆与持久化
- 所有对话接口返回 `Flux<String>`（SSE 打字机效果），前端 React 用 EventSource/fetch stream 渲染。
- 对话记忆用滑动窗口（如最近 20 条），但 **Repository 从内存换成 `JdbcChatMemoryRepository`（落库 MySQL）**——原项目用 `InMemoryChatMemoryRepository`，**重启即丢**；新方案重启后记忆仍在。
- 会话与消息同样存 MySQL：会话表用 `chat_session`，消息表用 `chat_message`，不再依赖 `InMemoryChatHistoryRepository`。
- 记忆与历史的存取键从单一 `chatId` 升级为 **userId + type + chatId**，实现按用户隔离并避免三种模块复用同一 `chatId` 时串台。

### 5. 用户管理与数据隔离（新增模块 ④）
- **认证**：Spring Security + JWT。注册（密码 BCrypt 加密存 MySQL 用户表）→ 登录校验后签发 JWT → 前端持有 token，后续请求经过滤器校验。
- **鉴权范围**：除 `/auth/register`、`/auth/login` 外，所有 `/ai/*` 接口需有效 token；从 token 解析出当前 `userId`。
- **数据隔离**：用户表 + 会话、会话历史、对话记忆、上传的 PDF/向量 全部挂 `userId`；查询一律按当前 userId 过滤，A 用户看不到 B 用户的会话与文档。`chatId` 不再由前端随意指定，需校验归属当前用户。
- **会话/消息隔离**：`chat_session` 用 `(userId, type, chatId)` 唯一定位；`chat_message` 通过 `sessionId` 逻辑关联到会话，并冗余 `userId` 做快速归属校验。
- **向量隔离**：PDF 入库时 metadata 同时带 `userId` + `chatId` + `filename`，检索时按 userId 过滤（Redis 向量库支持 metadata 过滤）。

### 6.5 语言
- **界面：全英文**（不做 UI 多语言切换）。
- **对话语言由用户决定**：用户让 Claude 用中文/英文回答即可（Claude 多语言能力天然支持）；可在系统提示中加"用与用户相同的语言回答"。
- embedding 选阿里 `text-embedding-v4`（多语言），中英文文档检索均可。

### 6. 关键集成注意点
- **双模型 Provider 共存**：Anthropic starter 负责 chat，OpenAI 兼容 starter（指向 DashScope）仅负责 embedding。需在配置中分别指定，避免 chat 自动用到 OpenAI 那套。
- Spring AI 与 Anthropic starter 的版本兼容需在落地时核对（确认 `spring-ai-starter-model-anthropic` 与现有 Spring AI 版本匹配）。
- API key 用环境变量：`ANTHROPIC_API_KEY`、`DASHSCOPE/OPENAI_API_KEY`、AWS 凭证。

## 后端 API 契约（供 React 前端对接，沿用原路径风格）

除 `/auth/*` 外所有接口需带 JWT；后端从 token 取 userId，数据按 userId 隔离（chatId 校验归属）。

| 端点 | 方法 | 鉴权 | 说明 |
|------|------|------|------|
| `/auth/register` | POST | 否 | 注册（用户名 + 密码） |
| `/auth/login` | POST | 否 | 登录，返回 JWT |
| `/ai/chat` | GET/POST(流) | 是 | 多模态对话（prompt, chatId, 可选 files=图片） |
| `/ai/history/{type}` | GET | 是 | 当前用户某类型的会话列表（返回 `id + chatId + title`） |
| `/ai/history/{type}/{chatId}` | GET | 是 | 某会话历史消息（读 `chat_message`，后续可带附件） |
| `/ai/pdf/upload/{chatId}` | POST | 是 | 上传 PDF：传 S3 + 记元数据 + 建向量库，返回 fileId |
| `/ai/pdf/file/{fileId}` | GET | 是 | 校验归属后返回 S3 预签名 URL（PDF 预览），前端用该 URL 加载 |
| `/ai/files/{fileId}` | GET | 是 | 校验归属后返回 S3 预签名 URL（图片等），前端 `<img>` 用该 URL 显示 |
| `/ai/pdf/chat` | GET(流) | 是 | 基于 PDF 的 RAG 问答 |
| `/ai/travel` | GET(流) | 是 | 旅游推荐助手（带工具调用：天气/景点/汇率/收藏行程） |

## 分阶段实施计划

**阶段 0 — 脚手架**
- 新建 Spring Boot + Spring AI 工程；加入 `spring-ai-starter-model-anthropic`、OpenAI 兼容 starter（embedding）、`spring-ai-starter-vector-store-redis`、PDF reader、Spring Security + JWT、`spring-cloud-aws-starter-s3` 依赖。
- Docker 起 Redis Stack（`redis/redis-stack`）；MySQL 本地自建；配置 AWS S3 凭证/region/bucket（放 `application-local.yaml`）。
- 注意：原项目用 `autoconfigure.exclude` 禁用了 Redis 向量库自动配置，新项目要**移除该排除**以启用 `RedisVectorStore`。

**阶段 1 — 用户管理(④) + 鉴权**
- 用户表；`/auth/register`（BCrypt）`/auth/login`（发 JWT）；Spring Security 过滤器校验 token；后续接口从 token 取 userId。

**阶段 2 — AI 对话(②) + 流式 + 持久化记忆**
- Claude ChatClient bean + 系统提示词；`/ai/chat` 流式；图片多模态；对话记忆用 `JdbcChatMemoryRepository` 落库 MySQL（重启不丢），按 `userId+type+chatId` 隔离。
- 会话表 `chat_session` + 消息表 `chat_message`：前端会话列表和历史消息都读业务表，不再直接读 `ChatMemory`。

**阶段 3 — ChatPDF(①) RAG + 文件存储**
- 先补文件层：`stored_file` + `chat_message_attachment`，让图片/PDF 都能和具体消息绑定并在历史接口回放。
- PDF 上传：原文件传 S3 私有桶 `uploads/{userId}/`，元数据入 MySQL；解析/切块 → 阿里 embedding → Redis 向量库；取文件接口返回预签名 URL；`QuestionAnswerAdvisor` 检索；`/ai/pdf/*` 端点；文档按 `userId+chatId` 隔离（向量 metadata 带 `userId`）。

**阶段 4 — 旅游推荐助手(③) Function Calling**
- 定义工具：天气(Open-Meteo)/景点(OpenTripMap)/攻略(Wikivoyage)/国家(REST Countries)/汇率(Frankfurter)/收藏行程(MySQL)；旅游助手系统提示词；`/ai/travel` 流式 + 工具循环（多工具串联）。
- 申请 OpenTripMap 免费 Key（其余 API 无需 Key）。

**阶段 5 — React 前端**
- 登录/注册页 + 路由守卫（未登录跳登录）；三个 AI 模块页面：对话、ChatPDF（PDF 预览 + 问答）、客服；SSE 流式渲染；历史会话；Markdown 渲染；请求统一带 JWT。
- 界面全英文（无 UI 语言切换）。

**阶段 6 — 收尾**
- 会话历史落库、错误处理、nginx 部署（前端静态 + 代理后端）。

## 验证方式（端到端）
- **对话/图片**：`/ai/chat` 发文字与图片，确认 Claude 流式返回且能描述图片内容。
- **RAG**：上传一份中文/英文 PDF，提问只在文档中出现的细节，确认答案来自原文（可对比未上传时的回答）。
- **Function Calling**：向旅游助手提"下月去东京玩 4 天，推荐行程、看天气、再帮我存下来"，确认 Claude 串联调用 getWeather/searchAttractions/getExchangeRate/saveItinerary 等工具并综合输出，行程已落库。
- **多语言**：中英文 PDF 与提问各测一遍，验证 embedding 多语言检索效果。
- **用户隔离**：注册 A、B 两个用户；A 上传 PDF、产生会话后，用 B 登录确认看不到 A 的会话与文档。
- **记忆持久化**：对话几轮后重启后端，确认历史与上下文记忆仍在。
- 用 `response.usage` / 日志确认走的是 `claude-opus-4-8`。

## 待确认 / 风险
- **向量库**：定为 Redis Stack（Docker 起库）。须用 Redis Stack（含 RediSearch），普通 Redis 不支持向量检索；并移除原项目对 Redis 向量库自动配置的 `exclude`。
